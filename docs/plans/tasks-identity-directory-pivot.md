- [x] **T27 (P1, human: ~2-3d / CC: ~3-4h)** — identity — phone verification gates access to the app itself, Signal-style (universal anti-spam gate)
  - **Decision 2026-07-14 (founder):** like Signal, you cannot get past onboarding without registering and verifying a phone number. There is no unverified-but-usable app state — no pairing, no messaging, no directory. Onboarding order: verify phone → claim username → app unlocks. Today verification only means search inclusion + a looser rate-limit tier (T2/T20 semantics); that's superseded.
  - **App-side: DONE** (shipped in the "rework UI" commit, before this review formally checked it off). `app/.../onboarding/Onboarding.kt` + `App.kt` hard-gate on signup+verify+username-claim before `ChatEngine` is ever constructed. `session` stays null (no engine, no pairing, no messaging) until onboarding completes.
  - **Transport-side design, resolved 2026-07-16 (`/plan-eng-review`), NOT YET BUILT:** founder was explicit that "the entire app" — including messaging, not just directory access — must be gated, so client-gate-only (accepting a custom-client bypass) was rejected. Chosen mechanism:
    - **Directory-issued single-use attestation tokens.** After phone verification, directory hands the client a batch of N Ed25519-signed opaque tokens (no `user_id` embedded — stays unlinkable, relay never learns identity). Client spends one token per `create_mailbox`/`create_group_inbox` call.
    - **Relay verifies offline** against a cached directory pubkey (pure signature check, no live call to directory — preserves T21's crash-isolation guarantee exactly). Spent-token tracking is **pruned by the token's own embedded expiry, not by FIFO count** (caught by outside-voice review: count-based eviction, copied naively from `state.rs`'s PoW-challenge pattern, would let an evicted-but-still-valid token be replayed — fail-open. Expiry-based pruning is safe because an expired entry's token would fail its own expiry check regardless of spent-tracking).
    - **Key distribution: baked into relay config at deploy time** (env var, same pattern as `DIRECTORY_ALLOW_DEV_OTP_VENDOR`/Twilio creds), never fetched over the network. Zero new relay→directory runtime coupling. Rotation is an ops action (config update + restart), not a code change — needs an overlap window in the rotation runbook so old-key-signed unexpired token batches don't fail mid-rotation.
    - Client re-fetches a new batch (authenticated, needs `sessionToken`) when running low.
  - **T2 soft-gate bug found + resolved:** `directory/src/verify.rs`'s `VerificationOutcome::Degraded` (vendor timeout) still returns a usable `session_token` from `POST /verify` (`directory/src/api.rs`), and `Onboarding.kt` never checked `result.verified` — a vendor outage today lets an unverified user straight into the app, already violating "no unverified app state" in shipped code. **Fix (client-side, small):** check `result.verified` in `Onboarding.kt`; `false` shows a retry/resend error, no transition to `ClaimUsername`. No new grace-window state machine. Directory-side `Degraded` accounts still exist in the DB unchanged (T2 code itself doesn't change).
  - **BUILT 2026-07-16 (`/plan-design-review`).** The gate is closed and tested. Three corrections to the description above, found while building it:
    1. **The soft-gate bug was worse than "lets an unverified user in".** `App.kt:42-46` persists the session on completion (`store.save`), and `store.load()` is the only thing that decides whether onboarding re-runs — so a vendor outage let an unverified user in **permanently**, across restarts. Not a second bug: `onComplete` is only reachable *through* the verify transition, so gating that transition closes the persistence symptom too. One fix, two symptoms.
    2. **"Shows a retry/resend error" was the wrong design and is not what shipped.** A `Degraded` outcome is NOT a wrong code — `CodeRejected` is a 400 that already arrives as a `DirectoryException`. Reaching `verified == false` means the user typed the **right** code and *our* vendor was unreachable. **ARCH-1 — ~~[P1] `Degraded` is near-unreachable because no timeout is

**It is backwards.** reqwest's *blocking* client is the one with a default

**ARCH-5 [P0] (confidence: 9/10, outside voice — the finding this whole review
should have led with) `/verify` is a fail-open authentication bypass.**

Verified line by line:
- `verify.rs:240-245` — on `Err(VendorError::Timeout)` the submitted code is **never validated**. `Ok(false) => CodeRejected` is unreachable on that branch, so *any* code yields `Degraded`.
- `api.rs:198-206` — `find_user_by_phone_hash(&hash)` → `Some(id) => id`: **the victim's existing `user_id`**.
- `api.rs:211-216` — `mark_verified(user_id, false)`: **demotes an already-verified victim**.
- `api.rs:217` — `create_session(user_id)`: mints a live session **bound to the victim**.
- `api.rs:99-108` — `authenticate` is a pure sessions-map lookup and **never checks `verified`**.

During any Twilio slowdown >30s, an unauthenticated attacker POSTs
`{"phone":"<victim>","code":"anything"}` and receives an authorized session as
that user. With it: `DELETE /account` (`api.rs:405-416`) erases them;
`POST /pairing-bootstrap` (`api.rs:338-353`) **replaces the victim's published
KeyPackage with the attacker's**, so the next person who searches for the victim
and consumes the bootstrap pairs with the attacker — **an MLS pairing MITM
reachable through a vendor timeout**. The precondition is manufacturable:
`/verify` is unauthenticated and unthrottled and each call pins a worker for up
to 30s, so a flood produces the timeouts the flood then exploits.

**This reframes T27 entirely.** The task doc files the custom-client bypass as a
*transport* problem for attestation tokens to solve later. It is not. It is a
*directory* problem, live in the very endpoint this diff is about. DT1's
client-side gate buys exactly one thing: an honest user during an outage sees a
chip instead of walking in. **It buys zero security** — the attacker runs curl,
not `Onboarding.kt`.

2. ~~**Sessions in Postgres.**~~ **RESOLVED 2026-07-17 — and it was a bug, not a
   preference.** Filed as an architectural tidy-up ("three findings point at the
   same fix"). Checking the premise before implementing turned it into a P0:
   `store.rs` justified the in-memory map with *"a directory restart means
   callers re-verify; that's an acceptable ceiling at this stage."* **Callers
   could not re-verify.** `App.kt:42` shows onboarding only when the stored
   session is null, the token is persisted (`SessionStore`), `clear()` had **zero
   call sites**, and no 401 was handled anywhere. So a restart did not send anyone
   back through verification — it left every installed client holding a token the
   server had forgotten. **One deploy of the directory permanently broke search
   and pairing for every user**, recoverable only by wiping app data.

   Shipped both halves:
   - **`0005_sessions.sql`** — a table with `created_at` (90-day TTL, so the map
     that "only grows" is now bounded) and `idx_sessions_user_id` (ET2's
     revocation was an O(n) scan only because a `token -> user_id` map has no
     reverse direction). `erase_user`'s `DELETE` moved *inside* the transaction,
     which the in-memory version could not do — it had to run after `commit`, or a
     rolled-back erasure would have locked out a live user. `session_user_id`
     joins `users` and re-checks `deleted_at`: erasure deletes the sessions, so
     that join should never matter, and a tombstone is exactly what a dangling
     token used to authenticate as.
   - **The 401 path** — `isSessionExpired()` clears the stored session and
     returns to onboarding. The table removes the *routine* cause; TTL expiry,
     ET2 revocation, and a restored database remain, and without a handler any one
     of them is the same permanent dead end, just rarer.

   Tested at the root: `a_token_survives_the_process_that_issued_it` builds a
   second `DirectoryStore` on the same pool — which is all a restart ever was —
   and asserts the token still resolves.
